# 通用分层文档规范

> 状态：草案 v0.1  
> 最近更新：2026-06-02  
> 适用范围：使用 AI 协作开发的软件项目文档体系，包括项目说明、变更计划、设计、实现记录、测试用例索引与证据报告。  
> 核心目标：让人和 AI 都能用最少上下文找到当前事实，同时保留历史需求和变更原因，避免过期文档误导后续开发。

---

## AI Quick Path

当 AI 接到开发、修复、重构或排查任务时，优先按这个顺序读文档，避免全量扫描：

1. `docs/overview.md`：确认系统是什么、边界是什么；
2. `docs/roadmap.md`：确认当前阶段、里程碑、优先级和风险；
3. `docs/plans/index.md`：按 `Type`、状态、完成度、模块、简介、阻塞项和下一步定位相关 plan；
4. 相关 plan 的 `requirements.md`：确认需求、边界和“不做什么”；
5. 相关 plan 的 `design.md`：确认预期实现方式；
6. 相关 plan 的 `implementation.md`：确认真实落地、偏差、验证和遗留问题；
7. 相关 plan 的 `test-cases.md` 与 `docs/test-cases/index.md`：确认测试覆盖和过期状态；
8. `docs/reports/`：仅在需要证据、日志、审计输出或测试报告时下钻。

如果 `docs/plans/index.md` 或文档元信息显示 `superseded` / `obsolete`，必须先跳到 `Replaced by` 指向的新文档。

---

## 1. 为什么要有这套规范

在大量使用 AI 辅助开发的项目里，AI 的主要成本不是“读一份长文档”，而是：

1. 不知道该先读哪一份；
2. 读到了过期 plan，却误以为它是当前事实；
3. 找不到某个测试用例来自哪个需求；
4. 变更改坏旧功能后，旧功能文档和全局测试索引没有被回写；
5. 功能实现和原始需求发生偏差，但偏差只存在于聊天记录或 commit 里。

因此文档体系必须分层：

- **索引层要薄**：只告诉 AI “现在是什么状态、下一步读哪里”；
- **计划层要完整**：讲清楚每个变更的需求、设计、实际实现和验证；
- **证据层要可追溯**：测试报告、SQL 备份、审计 CSV 等作为事实证据，但不承担入口说明职责。

一句话原则：**全局文档负责导航，plan 文档负责决策，implementation 文档负责事实，测试索引负责防漂移。**

## 2. 顶层文档结构

推荐目标结构如下：

```text
docs/
├── overview.md                         # 项目介绍：系统是什么、边界是什么、主要模块是什么
├── operation-manual.md                 # 用户手册：面向业务用户的当前操作方式
├── roadmap.md                          # 全局路线图：当前阶段、里程碑、重要入口
├── 通用分层文档规范.md                 # 本规范
├── test-cases/
│   ├── index.md                        # 全局测试用例主索引
│   ├── by-layer.md                     # 按项目测试分层索引
│   └── by-module.md                    # 按业务模块 / 里程碑 / 测试专项索引
├── plans/
│   ├── index.md                        # 变更 plan 总索引
│   └── YYYYMMDD-cNNN-short-name/
│       ├── requirements.md
│       ├── design.md
│       ├── implementation.md
│       └── test-cases.md
└── reports/
    └── ...                             # 测试报告、审计报告、备份 SQL、脚本输出
```

### 落地兼容规则

已有项目通常会存在大量历史单文件 plan，例如按月份、版本、迭代或专题归档的文档。这些文件不需要一次性迁移。

从本规范生效后：

- **新建复杂变更**优先使用 `docs/plans/YYYYMMDD-cNNN-short-name/` 四文档结构；
- **变更类型用元信息区分**，例如 `feature`、`bugfix`、`refactor`、`test-infrastructure`、`docs`、`ops`；
- **历史单文件 plan**保持原位；下次被功能变更或重构影响时，再补充“已被替代 / 已受影响 / 新 plan 链接”；
- `docs/overview.md` 只写项目当前事实入口；`docs/roadmap.md` 继续承担阶段、里程碑和计划索引职责。

### 变更类型与目录策略

通用规范不强制把 feature、bugfix、refactor、test-infrastructure 拆成不同物理目录。推荐默认做法是：**所有变更 plan 放在同一个 `docs/plans/` 下，用 `Type` 字段和索引视图区分**。

推荐类型：

| Type                  | 用途                                                                      | 典型特点                                 |
| --------------------- | ------------------------------------------------------------------------- | ---------------------------------------- |
| `feature`             | 新能力、新流程、新页面、新接口                                            | 需要完整需求边界、设计、实现、测试       |
| `bugfix`              | 修复现有行为缺陷                                                          | 必须记录复现、根因、回归用例             |
| `refactor`            | 以结构调整为主，原则上不改变外部行为                                      | 必须记录影响面、保持不变的行为、回归测试 |
| `test-infrastructure` | 测试框架、测试数据、fixture、CI、测试报告机制、历史补覆盖、跨模块测试专项 | 不属于某个单一业务功能，但仍是变更 plan  |
| `docs`                | 文档规范、用户手册、架构说明等                                            | 必须说明是否改变事实入口或流程           |
| `ops`                 | 部署、环境、运维、脚本、监控告警等                                        | 必须说明环境影响和回滚方式               |

如果项目规模较大，可以在项目适配层选择拆目录，例如：

```text
docs/plans/features/
docs/plans/refactor/
docs/plans/test-infrastructure/
```

但拆目录只是检索优化，不改变规则：**每个变更都必须有同一套需求、设计、实现、测试和索引回写机制。**

### `docs/plans/index.md` 主索引

如果不使用物理目录按 Type 分类，`docs/plans/index.md` 就是必须维护的主入口。它不是可选目录清单，而是 AI 的第一层检索表：AI 应该先读它，再决定下钻哪个 plan。

主索引每个 plan 至少一行，必须包含简介、完成度、阻塞项和下一步：

| Change ID      | Type    | 标题     | 简介                             | 状态        | 完成度 | 阻塞项     | 下一步         | 模块 / 里程碑 | Owner  | Plan                     | 最近更新   | 替代关系 |
| -------------- | ------- | -------- | -------------------------------- | ----------- | ------ | ---------- | -------------- | ------------- | ------ | ------------------------ | ---------- | -------- |
| C-20260601-001 | feature | 示例能力 | 一句话说明要解决的问题和当前边界 | in-progress | 60%    | 等接口确认 | 补 L2 集成测试 | 模块 A        | team-a | `20260601-c001-example/` | 2026-06-01 | -        |

字段含义：

- **Change ID**：稳定编号，不能因为目录移动或标题变化而改变；
- **Type**：`feature` / `bugfix` / `refactor` / `test-infrastructure` / `docs` / `ops` 等；
- **简介**：1-2 句话，说明这个 plan 解决什么问题、当前边界是什么；
- **状态**：使用第 3 节状态机；
- **完成度**：建议用百分比或 checklist 比例，例如 `60%`、`3/5`；必须能从 `implementation.md` 或任务清单解释；
- **阻塞项**：当前不能继续推进的原因；没有则写 `-`；
- **下一步**：最小可执行动作，例如“补 L2 测试”“等接口确认”“回写旧文档”；
- **模块 / 里程碑**：用于生成模块视图和 roadmap；
- **Plan**：指向目录或主文档；
- **替代关系**：如果该 plan 替代旧 plan 或已被新 plan 替代，必须写链接。

主索引可以按 Type 增加视图小节，但不要拆散事实来源。推荐结构：

```markdown
# Plans Index

## All Changes

| Change ID | Type | 标题 | 简介 | 状态 | 完成度 | 阻塞项 | 下一步 | 模块 / 里程碑 | Owner | Plan | 最近更新 | 替代关系 |
| --------- | ---- | ---- | ---- | ---- | ------ | ------ | ------ | ------------- | ----- | ---- | -------- | -------- |

## By Type

### feature

| Change ID | 标题 | 状态 | 完成度 | 阻塞项 | 下一步 | Plan |

### bugfix

| Change ID | 标题 | 状态 | 完成度 | 阻塞项 | 下一步 | Plan |

### refactor

| Change ID | 标题 | 状态 | 完成度 | 阻塞项 | 下一步 | Plan |
```

维护规则：

- 新建 plan 时，先登记 `docs/plans/index.md`，再写四文档；
- 状态、完成度、标题、阻塞项、下一步、替代关系变化时，同步更新主索引；
- 顶层 `docs/roadmap.md` 可以引用主索引，但不要复制每个 plan 的长细节；
- 如果项目额外维护 `by-type.md`、`by-module.md` 等视图，它们只能派生自主索引，不能成为另一份事实来源。

### 测试在整体规范中的位置

测试不是独立于开发文档之外的另一套主线。测试是每个变更 plan 的组成部分：

- 变更的 `requirements.md` 写验收口径；
- 变更的 `design.md` 写测试设计；
- 变更的 `implementation.md` 写真实验证记录；
- 变更的 `test-cases.md` 写本变更新增、修改、保留、作废的用例；
- `docs/test-cases/` 只做全局反向索引，方便从测试用例查回需求和变更原因。

`test-infrastructure` 类型只用于测试体系本身、历史补覆盖或跨模块测试专项，不替代普通业务变更里的测试章节。

## 3. 状态机

每份 plan 文档都必须显式标注状态。不要只写“完成了”，要写清楚它是否仍代表当前事实。

### 3.1 Plan 状态

| 状态                              | 含义                       | AI 使用方式                  |
| --------------------------------- | -------------------------- | ---------------------------- |
| `draft` 草案                      | 还在讨论，不能作为开发依据 | 只能参考背景，不能直接实现   |
| `approved` 已确认                 | 需求和设计已确认，尚未实现 | 可以作为实现依据             |
| `in-progress` 实施中              | 正在开发，内容可能变化     | 实现时必须同步回填           |
| `implemented` 已实现              | 代码已落地，但未完整验证   | 读 implementation 与测试记录 |
| `verified` 已验证                 | 实现和测试均已确认         | 可作为当前事实               |
| `superseded` 已被替代             | 有新 plan 接管此功能       | 必须跳到 `Replaced by`       |
| `deprecated` 已过时但仍有历史价值 | 不再代表当前行为           | 只能用于追溯原因             |
| `obsolete` 已作废                 | 功能或测试已失效           | 不得作为实现依据             |

推荐在每份文档开头使用统一元信息块：

```markdown
> Status: approved
> Stable ID: C-20260601-001
> Type: feature
> Owner: domain-or-team
> Created: 2026-06-01
> Last reviewed: 2026-06-01
> Progress: 60%
> Blocked by: none
> Next action: 补 L2 集成测试
> Replaces: none
> Replaced by: none
> Related plans: none
> Related tests: TC-DOMAIN-CHANGE-001, TC-DOMAIN-CHANGE-002
```

### 3.2 Test Case 状态

测试用例状态只用于 Case，不直接替代 Plan 状态。

| 状态         | 含义                             | AI 使用方式                         |
| ------------ | -------------------------------- | ----------------------------------- |
| `active`     | 当前有效，应该继续维护           | 可作为当前测试依据                  |
| `stale`      | 可能过时，尚未确认               | 不能直接作为结论，需核对 owner plan |
| `deprecated` | 不推荐新增依赖，但历史上仍可解释 | 仅用于追溯或迁移                    |
| `obsolete`   | 已失效，不应再执行或作为当前依据 | 必须查看作废原因和替代用例          |

### 强制规则

- 行为发生变化时，不能静默修改旧需求为“当前样子”。必须新增或更新对应 plan，并在旧文档标注 `Replaced by`。
- 文档被后续变更影响时，不能只改新 plan。必须回写到受影响旧文档的“变更记录 / 受影响记录”。
- 测试用例作废时不能删除。必须在全局测试索引中标注 `obsolete`，并写明由哪个变更导致。

## 4. 变更 plan 四文档结构

每个复杂变更目录命名：

```text
docs/plans/YYYYMMDD-cNNN-short-name/
```

示例：

```text
docs/plans/20260601-c001-approval-status-amount-standard/
```

`cNNN` 是当日或当前阶段内的人类可读编号，用于避免同一天多个变更重名。`short-name` 使用英文或拼音短名，便于终端检索；标题中可以使用中文。

### 4.1 `requirements.md` 需求文档

目标：用白话讲清楚“为什么做、做什么、不做什么、边界在哪里”。

必须包含：

- **背景**：触发原因、业务问题、用户反馈或测试发现；
- **用户故事 / 使用场景**：谁在什么情况下需要这个能力；
- **要做什么**：按业务语言描述，不写代码细节；
- **不做什么**：明确排除项，防止后续误扩范围；
- **业务规则**：状态、金额、权限、时间、数据范围等可判定规则；
- **边界与异常**：空值、重复、并发、权限不足、跨组织 / 租户 / 工作区、历史数据；
- **验收口径**：什么情况算完成；
- **开放问题**：尚未拍板的点，必须有负责人和日期；
- **变更历史**：需求发生变化时追加记录。

最重要的部分不是“要做什么”，而是“**不做什么**”。AI 最容易在边界不清时主动扩写功能。

### 4.2 `design.md` 设计文档

目标：给 AI 开发提供代码层面的实现指导。

必须包含：

- **总体方案**：后端、前端、数据库、测试的切分；
- **涉及模块与文件**：预期新增 / 修改 / 不动的文件或目录；
- **接口设计**：服务接口、API client、请求响应、错误码；
- **数据设计**：表、字段、索引、字典、migration；
- **权限与组织边界**：角色、功能权限、数据范围、组织 / 租户 / 工作区隔离；
- **状态机 / 口径**：枚举值、聚合规则、生命周期；
- **兼容策略**：历史数据、旧接口、灰度、回滚；
- **测试设计**：需要覆盖哪些 L1-L5 测试；
- **风险与替代方案**：为什么选当前方案，放弃了哪些方案。

设计文档允许在实现前被修改，但实现开始后，重大设计变化必须同步记录到 `implementation.md` 的偏差部分。

### 4.3 `implementation.md` 实现文档

目标：记录真实落地方式，而不是重复设计文档。

必须包含：

- **最终改动清单**：按后端 / 前端 / DB / 测试 / 文档分类；
- **实际涉及文件**：最好带相对路径；
- **与设计的偏差**：哪里没按设计做，为什么；
- **踩坑与修复**：错误现象、根因、最终处理；
- **数据处理**：migration、补丁 SQL、备份、回滚说明；
- **部署与重启要求**：哪些服务需要重启、哪些缓存需要清理；
- **验证记录**：跑了哪些命令、哪些测试通过或失败；
- **遗留问题**：明确是否阻塞上线。

如果实现和需求发生冲突，以 `implementation.md` 记录为事实，但必须回头修正 `requirements.md` 或新增替代 plan，不能让两者长期互相矛盾。

### 4.4 `test-cases.md` 测试用例文档

目标：保存本变更的局部测试用例，并供全局测试索引反向引用。

推荐表格：

| Case ID              | 标题             | 层级 | 类型       | 前置条件               | 步骤                 | 期望                 | 自动化路径            | 状态   |
| -------------------- | ---------------- | ---- | ---------- | ---------------------- | -------------------- | -------------------- | --------------------- | ------ |
| TC-DOMAIN-CHANGE-001 | 关键业务规则回归 | L5   | regression | 已有满足前置状态的数据 | 执行触发该规则的操作 | 系统按规则拦截或放行 | `path/to/e2e.spec.ts` | active |

字段说明：

- **Case ID**：稳定 ID，不随文件名变化；
- **层级**：遵循当前项目测试规范定义的分层，例如 L1-L5；
- **类型**：`unit` / `integration` / `e2e` / `regression` / `contract` / `manual`；
- **自动化路径**：没有自动化时写 `manual`，不能留空；
- **状态**：`active` / `stale` / `deprecated` / `obsolete`；
- **作废原因**：如果不是 active，必须说明由哪个变更导致。

## 5. 特殊变更类型的补充要求

所有变更默认使用第 4 节的四文档结构。不同 `Type` 可以增加额外关注点，但不必改变目录结构。

### 5.1 `refactor` 类型

重构不是“低配文档”。它也必须有需求、设计、实现、测试，只是重点不同：必须额外强调影响面和保持不变的行为。

`requirements.md` 必须回答：

- 触发因素是什么：bug、复杂度、权限漏洞、测试覆盖缺口、性能问题；
- 为什么这是重构而不是业务功能；
- 影响哪些原功能、页面、接口、测试；
- 哪些功能明确不应该改变；
- 是否需要用户感知的操作手册变更；
- 验收时如何证明“功能没变”或“只在预期范围内变了”。

建议增加影响表：

| 受影响对象   | 原文档                           | 影响类型       | 预期行为         | 需要回写 |
| ------------ | -------------------------------- | -------------- | ---------------- | -------- |
| 受影响功能 A | `docs/plans/.../requirements.md` | 口径或结构调整 | 保持指定行为不变 | 是       |

`design.md` 必须包含：

- 新旧结构对照；
- 迁移步骤；
- 兼容与回滚策略；
- 影响面验证方式；
- 需要新增、修改、作废的测试用例；
- 是否需要分阶段发布。

`implementation.md` 必须包含：

- 实际改动清单；
- 受影响功能文档是否已回写；
- 全局测试索引是否已更新；
- 哪些测试因重构作废、替换或新增；
- 与设计偏差；
- 验证结果。

`test-cases.md` 除了新测试，还要列出回归测试：

| Case ID              | 来源功能 | 用途                 | 状态     | 说明           |
| -------------------- | -------- | -------------------- | -------- | -------------- |
| TC-DOMAIN-CHANGE-001 | 功能 A   | 防止关键业务规则回退 | active   | 重构后继续保留 |
| TC-OLD-001           | 历史实现 | 已被新设计替代       | obsolete | R2-03 导致作废 |

大型重构可选 `impact-map.md`：

如果一个重构影响 3 个以上功能、10 个以上测试用例，或跨多个子工程，建议新增 `impact-map.md`，专门维护影响矩阵。小重构不强制新增第五份文档，避免文档数量膨胀。

### 5.2 `bugfix` 类型

`bugfix` 类型必须额外记录：

- 复现步骤；
- 预期行为与实际行为；
- 根因分析；
- 最小修复范围；
- 防复发测试用例。

### 5.3 `test-infrastructure` 类型

`test-infrastructure` 类型必须额外记录：

- 影响哪些测试层级、模块或 CI 流程；
- 是否改变测试数据、fixture、环境隔离或报告口径；
- 是否是历史补覆盖；
- 如果补测对象对应已有变更 plan，必须链接原 plan；
- 新增 Case ID 必须同步到 `docs/test-cases/`。

## 6. 全局测试用例索引

位置：

```text
docs/test-cases/
```

作用：从“测试用例”反向查到“需求 / plan / 重构原因 / 自动化文件 / 当前是否有效”。

全局测试用例索引分为一个主索引和两个视图索引：

| 文件                           | 维度                 | 用途                                                               |
| ------------------------------ | -------------------- | ------------------------------------------------------------------ |
| `docs/test-cases/index.md`     | 主索引               | 保存 Case ID、owner plan、自动化路径、状态、最后验证时间等事实字段 |
| `docs/test-cases/by-layer.md`  | 测试层级             | 按项目测试分层查找用例，例如 L1-L5                                 |
| `docs/test-cases/by-module.md` | 模块 / 里程碑 / 专项 | 按业务模块、系统模块、里程碑或测试专项查找用例                     |

这两个视图是正交的：同一个 Case ID 必须能同时从“层级”和“模块 / 里程碑 / 专项”两个方向被找到。

推荐字段：

| Case ID              | 标题             | 所属功能 | Owner Plan                     | 层级 | 自动化路径            | 状态     | 最后验证   | 作废 / 替代原因              |
| -------------------- | ---------------- | -------- | ------------------------------ | ---- | --------------------- | -------- | ---------- | ---------------------------- |
| TC-DOMAIN-CHANGE-001 | 关键业务规则回归 | 功能 A   | `docs/plans/20260601-c001-...` | L5   | `path/to/e2e.spec.ts` | active   | 2026-06-01 | -                            |
| TC-LEGACY-001        | 旧实现兼容用例   | 模块 B   | `docs/plans/20260601-c002-...` | L2   | `path/to/test`        | obsolete | 2026-06-01 | 被 C-20260601-002 新设计替代 |

### 双维度索引规则

- **Case ID 是唯一主键**：同一用例在 `index.md`、`by-layer.md`、`by-module.md` 中必须使用同一个 Case ID。
- **主索引保存事实**：状态、owner plan、自动化路径、最后验证时间以 `index.md` 为准。
- **分层索引回答“该补哪层测试”**：按项目测试规范定义的层级组织，例如单元、集成、组件、端到端等。
- **模块索引回答“该模块覆盖了什么”**：按业务模块、系统模块、版本里程碑或测试专项组织，跟项目自己的测试路线图保持一致。
- **一个用例只归一个主层级**：避免层级统计重复计数；如果端到端用例依赖集成测试数据准备，仍按端到端归类。
- **一个用例可以属于多个模块**：跨模块业务链路可以在 `by-module.md` 多处出现，但不能创建多个 Case ID。

### 状态规则

| 状态         | 含义                             |
| ------------ | -------------------------------- |
| `active`     | 当前有效，应该继续维护           |
| `stale`      | 可能过时，尚未确认               |
| `deprecated` | 不推荐新增依赖，但历史上仍可解释 |
| `obsolete`   | 已失效，不应再执行或作为当前依据 |

### 强制更新场景

以下情况必须更新全局测试索引：

- 新增自动化测试；
- 删除或重命名测试文件；
- 修改业务口径导致旧断言不再成立；
- 重构导致某批测试不再适用；
- 手工测试转为自动化测试；
- 测试失败被确认是需求变化，不是 bug。
- 用例所属层级或模块 / 里程碑 / 专项发生变化。

## 7. 全局路线图

`docs/roadmap.md` 是唯一默认路线图，负责回答“项目现在走到哪里、下一步是什么、重要入口在哪里”。它可以引用 `docs/plans/index.md`，但不复制每个 plan 的长细节。

全局路线图只保留：

- 当前项目概况；
- 大阶段、里程碑和横向轨道；
- 重要里程碑；
- 入口链接；
- 当前优先级、依赖和风险；
- 当前仍有效的高层结论。

不放：

- commit 级流水；
- 单个 bug 的完整分析；
- 大量测试步骤；
- 已被替代的旧需求细节。

推荐引用方式：

| 里程碑 | 目标       | 关联计划                             | 状态        | 风险             |
| ------ | ---------- | ------------------------------------ | ----------- | ---------------- |
| M1     | 示例里程碑 | `docs/plans/index.md#c-20260601-001` | in-progress | 依赖外部接口确认 |

`docs/plans/` 默认不再另设 `roadmap.md`。如果项目确实需要某个专题路线图，例如“大型迁移专项路线图”或“测试体系专项路线图”，应在项目适配层明确命名和用途，避免和全局 `roadmap.md`、`plans/index.md` 重叠。

## 8. 交叉链接规则

为了让 AI 低 token 下钻，所有链接必须双向维护。

### 变更影响旧功能

新变更 plan：

```markdown
> Replaces: docs/plans/20260501-c003-old-rule/requirements.md
```

旧功能 plan：

```markdown
> Status: superseded
> Replaced by: docs/plans/20260601-c001-new-rule/requirements.md
```

### 结构调整影响功能

变更 plan 写：

```markdown
## Impacted Features

| Feature | Impact               | Required doc update       |
| ------- | -------------------- | ------------------------- |
| 功能 A  | 关键业务规则口径调整 | requirements + test-cases |
```

受影响功能文档写：

```markdown
## Change Log

- 2026-06-01: 受 C-20260601-002 影响，关键业务规则口径由旧规则调整为新规则。
```

全局测试索引写：

```markdown
| TC-DOMAIN-CHANGE-001 | ... | active | ... | C-20260601-002 后继续有效 |
| TC-DOMAIN-CHANGE-OLD-001 | ... | obsolete | ... | C-20260601-002 口径变化导致作废 |
```

## 9. AI 阅读路径

当 AI 接到一个任务时，推荐按以下顺序读文档：

1. `docs/overview.md` 或 `docs/roadmap.md`：确认项目和当前阶段；
2. `docs/通用分层文档规范.md`：确认文档规则；
3. `docs/plans/index.md`：按 Type、状态、完成度、模块和简介定位相关 plan；
4. 相关变更 plan 的 `requirements.md`：确认需求和边界；
5. 相关 `design.md`：确认预期实现方式；
6. 相关 `implementation.md`：确认真实落地和偏差；
7. `test-cases.md` 与 `docs/test-cases/index.md`：确认测试覆盖和过期状态；
8. `docs/reports/`：只在需要证据、日志、审计输出时下钻。

AI 不应该优先全文扫描 `docs/reports/` 或所有历史 plan。索引层必须先告诉 AI 哪些文档值得读。

## 10. 开发流程中的文档更新点

### 10.1 新变更

1. 在 `docs/plans/index.md` 登记 Change ID、Type、简介、初始状态和 owner；
2. 创建变更 plan 目录；
3. 先写 `requirements.md`，明确做什么和不做什么；
4. 写 `design.md`，经过确认后再开发；
5. 开发过程中持续回填 `implementation.md`；
6. 新增或更新 `test-cases.md`；
7. 更新 `docs/test-cases/index.md`、`by-layer.md`、`by-module.md`；
8. 状态、完成度或替代关系变化时同步更新 `docs/plans/index.md`；
9. 必要时更新 `docs/roadmap.md` 和 `operation-manual.md`。

### 10.2 修改已有功能

1. 判断是否是行为变化；
2. 如果是行为变化，新增一个变更 plan 或在当前仍活跃的 plan 中追加变更记录；
3. 旧需求被替代时标注 `superseded`；
4. 更新对应测试用例状态；
5. 回写全局测试索引；
6. 更新 `docs/plans/index.md` 的状态、完成度和替代关系。

### 10.3 `refactor` 类型变更

1. 创建变更 plan，并在元信息中标注 `Type: refactor`；
2. 在 `requirements.md` 写清楚影响哪些功能和不改变哪些行为；
3. 在 `design.md` 写迁移策略和回归测试策略；
4. 实施后在 `implementation.md` 勾掉“功能文档回写”和“测试索引回写”；
5. 更新 `docs/plans/index.md`；如果影响全局阶段、里程碑或优先级，再更新 `docs/roadmap.md`；
6. 对作废测试标注 `obsolete`，不要直接删除。

### 10.4 `test-infrastructure` 类型变更

1. 创建变更 plan，并在元信息中标注 `Type: test-infrastructure`；
2. 适用场景：测试基础设施、历史补覆盖、跨模块测试专项；
3. 如果补测对象对应已有变更 plan，必须在测试专项文档中链接原 plan；
4. 如果补测发现原功能需求已经过时，必须回写原功能文档或新增替代 plan；
5. 补测新增的 Case ID 同样写入 `docs/test-cases/index.md`、`by-layer.md`、`by-module.md`；
6. 专项完成后，只在顶层 `docs/roadmap.md` 或项目适配层指定的专题路线图中记录阶段状态和入口，不复制所有测试细节。

### 10.5 项目适配层

本规范是通用底座。每个项目可以在 `docs/documentation-adapter.md` 或 `docs/README.md` 中补一页项目适配说明，只记录该项目的命名和分层选择，不改写通用规范本身。

建议适配项：

- 测试分层名称，例如 L1-L5、unit/integration/e2e，或其他项目自定义分层；
- 模块 / 里程碑划分，例如按业务域、版本、迭代、Stage、Epic 或系统模块；
- plan 编号规则，例如 `C-YYYYMMDD-NNN`、`R2-03`、`EPIC-123`；
- owner 命名规则，例如团队、业务域、服务名或代码 owner；
- 项目特有的用户手册、部署手册、运维手册入口；
- 历史文档的兼容目录和迁移节奏。

## 11. PR / 提交前检查清单

每次变更提交前，至少检查：

- [ ] 是否有对应变更 plan；
- [ ] `docs/plans/index.md` 是否登记了简介、状态和完成度；
- [ ] 需求边界是否写了“不做什么”；
- [ ] 实际实现是否回填到 `implementation.md`；
- [ ] 与设计不一致的地方是否有解释；
- [ ] 新增或修改的测试是否写入局部 `test-cases.md`；
- [ ] 全局测试索引 `index.md` / `by-layer.md` / `by-module.md` 是否同步更新；
- [ ] 旧测试作废时是否标注原因和替代项；
- [ ] 受影响的旧功能文档是否回写；
- [ ] `docs/plans/index.md` 的 Type、状态、完成度、替代关系是否同步；
- [ ] 用户手册是否需要更新；
- [ ] roadmap 是否只保留索引，不塞实现细节。

## 12. 常见反模式

避免以下写法：

- 把所有细节都塞进 `docs/roadmap.md`，导致 AI 每次都要读巨型文档；
- 旧需求被新需求替代后，直接把旧文档改成新口径，丢失历史原因；
- 测试用例只存在于测试文件名里，没有 Case ID；
- 删除过期测试，但不写明为什么过期；
- `refactor` 类型 plan 只写“清理代码”，不写影响哪些原功能；
- implementation 文档重复 design 文档，却不记录实际偏差；
- “TODO 后续补”没有日期、负责人和触发条件；
- 文档状态长期停在 `in-progress`，但代码已经合并。

---

## 13. 历史文档迁移建议

不要为了规范而一次性重写所有历史文档。迁移按“碰到再修”：

1. **新计划先规范**：从今天起的新复杂变更使用四文档结构；
2. **活跃计划优先拆分**：正在开发、仍会被 AI 高频读取的单文件 plan，优先迁移；
3. **旧计划只加状态**：历史价值大但不再变更的 plan，补 `Status`、`Replaced by`、`Related tests` 即可；
4. **测试索引渐进建设**：先收录新增测试和近期高风险测试，再逐步回填历史测试；
5. **报告不迁移**：`docs/reports/` 是证据层，不需要拆成 plan。
