# 通用测试规范

> 本规范是公司研发测试工作的**通用底座**，适用于所有采用「Spring Boot 后端 + Vue 前端」技术栈的项目。
> 各项目在自己仓库另写一份《项目测试落地说明》填充本规范留白的占位项（见 [§10](#10-落地与例外)），不重复抄通用内容。
>
> 本规范是**活文档**：执行中发现新坑、新约定，回写本文档。

## 1. 适用范围与目标

### 适用范围

- **技术栈**：Spring Boot（Java）后端 + Vue 3（TypeScript）前端。其他技术栈项目可参考本规范的**原则与流程**，分层工具自行替换。
- **对象**：所有研发人员、测试人员、技术负责人。
- **阶段**：从需求开发到提测、测试、缺陷修复、复测的全过程。

### 目标

1. **研发对自己代码的质量负第一责任**。提测前的低级缺陷应由研发自测拦截，不依赖测试人员兜底。
2. **测试人员聚焦深度验证**。把测试人员的精力从"重复发现研发本可自查的问题"释放出来，投入业务语义、探索性、集成与回归测试。
3. **缺陷不重复发生**。每个缺陷都沉淀为一个永久回归用例，同类问题不再复发。

> **一句话总纲**：研发自测兜住质量底线，测试人员做深度验证而非低级缺陷的清道夫；每个缺陷都进回归网。

## 2. 核心制度：自测 → 自测报告 → 测试准入

**这是本次规范确立的核心制度。** 研发人员必须完成自测并产出《自测报告》；测试人员收到自测报告、完成准入审查后，才允许开始测试。

### 2.1 流程总览

```
研发开发完成
   │
   ▼
研发自测（执行所有应跑的测试层 L1~L5）
   │ 不通过 ──▶ 研发自行修复 ──┐
   │ 全部应跑层全绿 + 覆盖率达标 │
   ▼ ◀──────────────────────────┘
填写《自测报告》
   │ 提测（附自测报告）
   ▼
测试准入审查（测试人员：完整性核对 + 抽查复现）
   │
   ├── 报告不全 / 覆盖率不达标 / 抽查复现失败 ──▶ 直接退回研发，不进入测试排期
   │
   ▼ 准入通过
测试人员正式测试
   │
   ├── 测出 bug ──▶ 退回研发：补 fail 用例复现 + 修复 + 重跑受影响层 + 更新自测报告
   │                          └──────────────────▶（重新提测，回到「测试准入审查」）
   │
   ▼ 无缺陷
测试通过 → 进入发布流程
```

### 2.2 研发职责

- **必须自测**：提测前执行本次改动涉及的**所有应跑测试层**（见 [§4](#4-测试分层模型五层)），不得"提测即甩锅"——把未自测的代码直接丢给测试人员属于流程违规。
- **产出自测报告**：按 [§3](#3-自测报告规范) 填写并提交《自测报告》，对其**真实性负责**。报告造假（声称跑过实际未跑、隐瞒已知问题）按严重流程违规处理。
- **如实披露**：已知未覆盖项、已知缺陷、跳过的测试层，必须在报告中显式列出，不得留白瞒报。
- **修复缺陷**：测试人员退回的缺陷，按 [§8](#8-缺陷处理规范) 补 fail 用例、修复、重跑受影响层、更新报告后重新提测。

### 2.3 测试职责

- **准入审查**：收到提测后，先做准入审查，**不直接开测**：
  1. **完整性核对**：自测报告字段是否齐全、应跑的层是否都跑了、覆盖率是否达标、命令输出/结果是否附齐。
  2. **抽查复现**：抽取若干关键用例本地复现，确认报告所述"全绿"属实。
- **不达标直接退回**：报告不全、覆盖率不达标、抽查复现失败的，**直接退回研发，不进入测试排期**。不替研发补测、不"边测边等研发补"。
- **深度验证**：准入通过后执行深度测试——业务语义、探索性、集成、回归、兼容性等研发自测难以覆盖的维度。
- **缺陷管理**：记录缺陷（可复现步骤、预期/实际、定级），退回研发，跟踪复测。

### 2.4 退回规则

| 退回时机       | 触发条件                                   | 处理                                                                                      |
| -------------- | ------------------------------------------ | ----------------------------------------------------------------------------------------- |
| **准入退回**   | 自测报告不全 / 覆盖率不达标 / 抽查复现失败 | 不进入测试排期，研发补齐后重新提测                                                        |
| **测试中退回** | 测试人员发现 bug                           | 研发按 [§8](#8-缺陷处理规范) 补 fail 用例 + 修复 + 更新报告，重新提测并**重新走准入审查** |

> **重新提测必须重走准入审查**：缺陷修复后重新提测的，自测报告必须更新（追加该缺陷的复现用例 + 重跑受影响层的结果），并重新经过准入审查。不存在"修个小 bug 就跳过审查"的快速通道。

### 2.5 角色与责任

| 角色           | 职责                                                                                    |
| -------------- | --------------------------------------------------------------------------------------- |
| **研发人员**   | 编写代码与各层测试；完成自测；填写并提交真实的自测报告；修复缺陷并补回归用例            |
| **测试人员**   | 准入审查（完整性核对 + 抽查复现）；不达标退回；执行深度/探索性/回归测试；记录与跟踪缺陷 |
| **技术负责人** | 审批例外申请（跳层、降低覆盖率门槛）；仲裁准入争议；定期审查覆盖率趋势与回归网健康度    |

## 3. 自测报告规范

自测报告是**独立交付物**：每个需求 / 提测任务对应一份独立的报告文档（建议放项目 `docs/` 下并提交 git，或作为 PR 附件），PR 描述里附链接与摘要。完整可复制模板见 [附录 A](#附录-a自测报告模板)。

### 3.1 必含字段

| 字段                    | 说明                                                                                                       |
| ----------------------- | ---------------------------------------------------------------------------------------------------------- |
| **基本信息**            | 需求/任务名称、关联 plan 或 issue、研发人、提测日期、代码分支与 commit                                     |
| **变更范围**            | 本次改动的模块/文件/接口清单（测试人员据此判断应跑哪些层）                                                 |
| **分层测试结果**        | 逐层（L1~L5）列：应跑/不适用、用例数、通过数、执行命令、结果（输出粘贴或截图）。**不适用的层必须注明原因** |
| **覆盖率**              | 各类别覆盖率数字 + 对应门槛 + 是否达标；**安全类代码单独列出并标注 100%**                                  |
| **反向验证**            | 做过哪些"故意打 bug"验证、确认对应用例确实会 fail（证明用例不是假阳性）                                    |
| **已知问题 / 未覆盖项** | 显式列出未覆盖的分支、已知缺陷、技术债；**不得留空瞒报**                                                   |
| **自测结论**            | 通过 / 不通过 + 研发签字（对报告真实性负责）                                                               |

### 3.2 「自测通过」的硬性定义

以下四条**缺一不可**，全部满足方可在报告中勾选"自测通过"：

1. **应跑的测试层全部执行且全绿**——无 skip、无 `@Ignore`/`.skip`、无被注释掉的用例。
2. **覆盖率达到对应分级门槛**（见 [§6](#6-覆盖率要求与统计)），安全类代码 100%。
3. **至少做过一次反向验证**——故意往被测逻辑打一个 bug，确认相关用例确实 fail，再还原。
4. **已知未覆盖项已在报告中显式列出**——允许有未覆盖项（经技术负责人批准的例外），但必须写明，不允许瞒报。

> 任一条不满足，自测结论只能填"不通过"，不得提测。

## 4. 测试分层模型（五层）

测试分**五层**——后端两层、前端三层。每层职责清晰、互补，不重复测同一件事。

```
L1 后端单元   纯逻辑，不启动应用、不连 DB              ── 数量最多、最快
L2 后端集成   启动应用 + 真实 DB + 缓存                ── 验证完整链路
L3 前端单元   纯函数 / composable / store，不 mount 组件
L4 前端组件   mount 组件，验渲染 + 交互状态机
L5 端到端     真浏览器 + 真后端                        ── 数量最少、最慢
```

### 4.1 各层定位

| 层     | 名称     | 工具                                           | 是否启动应用/DB        | 测什么                                                                                        |
| ------ | -------- | ---------------------------------------------- | ---------------------- | --------------------------------------------------------------------------------------------- |
| **L1** | 后端单元 | JUnit 5 + Mockito                              | 否                     | service 方法 / converter / util / 算法等纯逻辑，依赖全部 mock                                 |
| **L2** | 后端集成 | `@SpringBootTest` + 真实 DB + 真实缓存         | 是                     | 完整链路：接口 → service → DB；数据权限、事务、缓存、鉴权注解（如 `@PreAuthorize`）的真实行为 |
| **L3** | 前端单元 | Vitest（不 mount 组件）                        | 否                     | 纯函数 helper、`useXxx` composable 的边界、store 的 action/getter                             |
| **L4** | 前端组件 | Vitest + Vue Test Utils（mount 组件）          | 否（后端 API 全 mock） | 组件渲染 + 交互状态机；表单校验、弹窗/面板交互、父子联动                                      |
| **L5** | 端到端   | Playwright 或 Cypress + 真后端（测试 profile） | 是                     | 跨页面业务链路、真实鉴权、真实数据流；外部依赖走 mock                                         |

> **L3 与 L4 的区分**：是否 `mount` 组件。纯函数 / store 不 mount → L3；任何 `mount(Component, …)` → L4。两者共用 Vitest 配置，靠目录约定与报告归类区分。
>
> **L5 工具**：Playwright 或 Cypress 皆可，各项目按既有技术栈选型，在《项目测试落地说明》注明实际使用的框架；两者都要求"真浏览器 + 真后端（测试 profile）+ 连测试库"，对 L5 的分层定位与数据原则（§7.3）要求一致。

### 4.2 层比例参考

按代码量 / 执行时间的大致分布（金字塔，非硬指标）：

```
L1 后端单元   ~50%
L2 后端集成   ~20%
L4 前端组件   ~15%
L3 前端单元   ~10%
L5 端到端     ~5%
```

底层多而快、顶层少而慢。若发现 L5 用例数膨胀，多半是该下沉到 L1~L4 的逻辑被错误地堆到了 e2e。

### 4.3 改动类型 → 必加测试层

| 改动类型                                   | 必加层                                                                    |
| ------------------------------------------ | ------------------------------------------------------------------------- |
| 后端 service 新增/修改方法                 | L1；若涉及 DB / 缓存 / 事务，加 L2                                        |
| 后端 controller 新增/修改 endpoint         | L2（验鉴权注解 + 业务逻辑）                                               |
| 后端实体字段增/删                          | L2（验 mapper 查询字段对齐 + 相关业务链路重跑）                           |
| 数据权限 / 鉴权拦截逻辑                    | L2（覆盖各权限维度 + 反例：故意去掉鉴权应 fail）；**安全类，覆盖率 100%** |
| 前端 helper / util / composable            | L3（覆盖所有分支 + edge case）                                            |
| 前端 store action / getter                 | L3（mock 接口 + 验 state 变化）                                           |
| 前端组件 / 弹窗 / 关键交互                 | L4（验渲染 + 交互状态机）                                                 |
| 跨页面业务链路（如登录 → 切换 → 提交流程） | L5（真后端 + Playwright/Cypress）                                         |
| DB schema 变更 / 迁移脚本                  | L2（验字段对齐 + 业务链路重跑）                                           |
| 重构（行为不变的代码调整）                 | 复用既有各层用例验证行为不变；plan 标注新增/调整的 case 数                |

## 5. 测试计划规范（精确到 case）

测试计划必须**精确到 case**。粗粒度计划（停在"新建 xxx 模块的 e2e"这种模块名粒度）是历史漏覆盖事故的根因——禁止。

### 5.1 计划的最小粒度

- **逐 endpoint / 逐 service 方法展开**。每个 endpoint、每个 service 公共方法，至少列出三类 case：
  - **happy path**：正常输入、正常返回。
  - **边界**：空值、极值、最大/最小、临界条件、分页边界等。
  - **异常**：非法输入、权限不足、依赖失败、并发冲突等。
- **副作用分支清单**：凡是有副作用的方法（同步刷新其他表、发消息、撤销 token、清缓存、触发流程节点、写审计日志等），必须显式列出**每一个副作用分支**，每个分支至少配 1 个测试 case。只写"该 service 覆盖率 ≥ N%"不算数。
- **反例 case**：对鉴权、数据权限、关键校验，要有"故意破坏 → 应 fail"的反例 case，证明用例真的在守护。

### 5.2 计划评审

- 测试计划本身必须经过评审（研发互审或技术负责人评审），**确认 case 粒度足够细**才能开始编码。
- 评审重点：是否还停在模块名粒度？副作用分支是否列全？异常路径是否遗漏？

### 5.3 用例命名规范

- ✅ **功能化描述**：用例名直接说明"测什么、期望什么"。例：`doLogin_should_lockAccountAfter5Failures`、`permission-management-ui.spec.ts`。
- ❌ **禁内部代号**：`Phase3` / `StageN` / `R1-1` 这类项目内部编号——半年后无人看得懂。代号可进 Javadoc/注释，不进文件名与用例名。
- ❌ **禁无意义缩写**：`BasicApiPack` / `CrossMechanism` 这种看不出测什么的命名一律避免。
- 测试文件**按页面 / 模块 / 业务流程组织**，不按抽象"主题"跨页面塞进一个文件。

## 6. 覆盖率要求与统计

### 6.1 分级覆盖率门槛

覆盖率**分级要求**——安全相关代码必须 100%，业务核心高门槛，一般业务中门槛，框架代码不计。不追求 100% 全局覆盖率。

| 类别                         | 范围                                                                                     | 行覆盖   | 分支覆盖 | 强制时机     |
| ---------------------------- | ---------------------------------------------------------------------------------------- | -------- | -------- | ------------ |
| **安全类**                   | 认证登录、登录失败限制、验证码、token 校验、鉴权注解处理逻辑、数据权限处理、鉴权拦截器等 | **100%** | **100%** | **始终强制** |
| **业务核心**                 | 金额计算、状态机、权限/字段派发、关键算法、核心领域服务                                  | ≥ 85%    | ≥ 75%    | 强制         |
| **业务一般**                 | 常规 CRUD service / controller / 基础数据访问层                                          | ≥ 70%    | ≥ 60%    | 强制         |
| **框架 / 第三方 / 生成代码** | 框架原生包、第三方库、代码生成器产物                                                     | 不计     | 不计     | 排除         |

> **安全类清单**：每个项目维护一份《安全类代码清单》（package / 类名），新增安全相关代码时**必须同步更新该清单**。清单是安全类 100% 门槛的执行依据。

### 6.2 统计工具

| 层                        | 工具                                 | 报告             |
| ------------------------- | ------------------------------------ | ---------------- |
| L1 后端单元 + L2 后端集成 | JaCoCo（绑定到构建的 `verify` 阶段） | 各模块 HTML 报告 |
| L3 前端单元 + L4 前端组件 | Vitest 内置覆盖率（V8 / c8）         | lcov HTML 报告   |
| L5 端到端                 | **不计入覆盖率**                     | ——               |

> **为什么 L5 不计入覆盖率**：e2e 是"端到端正确性验证"，不是"代码行触达统计"的来源。把 e2e 计入会让行覆盖数字虚高，掩盖单测/集成测试的真实薄弱区。e2e 自身的指标是"通过率 / flaky 率"，不是 line coverage。

### 6.3 反向验证

覆盖率数字必须配反向验证才可信：

- 故意在被测逻辑里加一个**未被任何用例覆盖的分支** → 分支覆盖率应下降。
- 故意**删除一个安全类用例** → 安全类覆盖率应从 100% 掉下来。
- 若打了 bug / 删了用例覆盖率纹丝不动，说明覆盖率统计范围配置有误，须修正。

## 7. 测试数据与环境隔离

### 7.1 数据库分层（生产 / 开发 / 测试）

明确区分三类数据库，职责互不混用：

| 库         | 数据                      | 用途                                                   | 自动化测试                                                      |
| ---------- | ------------------------- | ------------------------------------------------------ | --------------------------------------------------------------- |
| **生产库** | 真实业务数据              | 线上运行                                               | **绝不连**——不读、不写、不测                                    |
| **开发库** | 生产库的**脱敏副本**      | 联调、**复现"生产数据相关"的问题**（数据形态贴近线上） | **不作为自动化测试数据源**；仅供人工 / 探索性复现，按需手工操作 |
| **测试库** | **仅测试 seed / fixture** | **专为自动化测试服务**                                 | 自动化测试**只连此库**；可全清、可重建                          |

- **自动化测试只连测试库**。"线上出了问题、要贴近生产数据形态去复现"时才用开发库（脱敏业务数据，非确定性 fixture），且不在其上跑自动化套件。
- 测试库的数据**全部为测试而生**，没有需要长期保护的业务数据——所以**允许清库重建**。这正是它区别于开发库 / 生产库的本质，也是下面 §7.3「干净起点」策略的前提。

### 7.2 环境隔离

- **独立测试环境**：测试使用**独立的数据库 schema、独立的缓存 db、独立的配置 profile**。
- **绝不连开发库 / 生产库跑测试**——避免污染、避免误删、避免环境间数据漂移。
- **凭据不入库**：测试环境的账号、密码、密钥只放在**不提交 git** 的本地配置文件中；提交到 git 的任何文档/脚本/配置都不得包含明文凭据。

### 7.3 测试数据原则

测试库专属、可重建，故采用「**干净起点 + 显式 seed**」模型，保证每次运行的一致性与确定性：

- **干净起点**：每个测试套件 / 测试运行前，把测试库重置到已知干净状态（清空 + 重新 seed），不依赖上一次运行的残留。因测试库无需保护共享业务数据，**允许全量清空 / 重建**（这与"共享数据库禁止 truncate 全表"的旧约束相反——前提正是测试库专为测试、可丢弃；见 §7.1）。
- **显式可重放 seed**：测试所需的基础数据（租户、角色、部门、用户、字典、菜单等）做成**显式、版本化、可重复执行的 seed**（SQL 脚本 / 代码 fixture），所有测试复用，不各自重复造。
- **精确断言**：断言针对自己 seed 的**已知数据**写精确期望值，**不写**"≥N 条""包含某条"这类依赖未知现存数据的松断言——松断言换个库就失效，且会掩盖问题。
- **per-test 隔离**：单个测试内部优先用**事务回滚**（如后端 `@Transactional` 测试方法跑完自动回滚）实现隔离、互不影响；跨事务的全栈 / e2e 流程，用"运行前重置 + 测试中造的临时数据打可识别标记（如特定 CREATOR 值）便于定位清理"保证可重复。
- **不依赖自增 ID**：测试数据的关键 ID 显式指定，不依赖数据库自增序列，避免环境 / 重放间错位。

### 7.4 外部依赖处理

- OCR、短信、第三方 API、SSO / CAS 等外部依赖，在测试环境**走 mock 或绕过**（如 `@Profile("test")` 的 mock controller）。
- 已确认业务代码未调用的外部依赖，无需处理——但要在《项目测试落地说明》中记录"确认未调用"的结论。

### 7.5 稳定性验收

- 关键 e2e / 集成测试套件应能**连续多次运行（如 100 次）不出现因数据残留导致的 flaky**。
- 首次因"上一个测试残留数据干扰"失败时，把残留表/数据加进清理清单。

## 8. 缺陷处理规范

### 8.1 bug fix 铁律

**任何缺陷修复，必须先有一个能复现该缺陷的 fail 用例。**

1. 先写一个用例**复现该 bug**，运行确认它**失败**。
2. 再修复代码。
3. 再运行确认该用例**通过**，且其他用例不回归。

没有复现用例的"裸修复"不允许提交——无法证明 bug 真被修掉，也无法防止复发。

### 8.2 QA 测出 bug 的回退流程

测试人员测出 bug 退回研发后，研发必须：

1. **补 fail 用例**：在合适的层（优先低层）写一个复现该 bug 的用例，确认失败。
2. **修复**代码。
3. **重跑受影响的所有测试层**，确认全绿、无回归。
4. **更新自测报告**：追加该缺陷的复现用例 + 重跑结果。
5. **重新提测**，回到 [§2.3](#23-测试职责) 的准入审查（重新走完整准入）。

### 8.3 回归网

- 所有缺陷的复现用例**永久保留**进测试套件，构成**回归网**。
- 同类缺陷再次出现时，回归网应能在研发自测阶段就拦截，不再流到测试人员。
- 定期审查回归网健康度（是否有用例被长期 skip、是否有 flaky 用例被静默忽略）。

### 8.4 缺陷定级（参考）

| 级别 | 含义                                 | 期望响应             |
| ---- | ------------------------------------ | -------------------- |
| 阻塞 | 核心功能不可用 / 数据错误 / 安全问题 | 立即修复，优先于一切 |
| 严重 | 主要功能受损，无合理 workaround      | 当迭代内修复         |
| 一般 | 功能缺陷，有 workaround              | 排期修复             |
| 轻微 | 文案 / 样式 / 体验小问题             | 可择期修复           |

> 具体定级标准与响应时限由各项目在《项目测试落地说明》中细化。

## 9. PR / commit 测试要求

- **每个 commit 至少带一个对应层的测试**：
  - 改后端 service / converter → L1
  - 改 controller / endpoint / 鉴权注解 / 数据权限 / SQL → L2
  - 改前端 helper / composable / store → L3
  - 改前端组件 / 弹窗 / 面板渲染或交互 → L4
  - 改跨页面业务链路 → L5
- **新 feature / 重构的 plan**：必须按 L1~L5 分别列出"新增 N 个 case"，完成后核对"N case 全过"。
- **PR 描述**：必须附《自测报告》链接 + 摘要。
- **PR review 必检项**：
  1. 对应层的测试是否齐全。
  2. 测试是否真的在本地 / CI 跑过（不是只写不跑）。
  3. 是否只测了 CRUD 的 200 响应、漏了**业务语义验证**（CRUD 产物 → 真实使用 → 验证生效）。
  4. service / 接口有改动是否**同步改了测试**——改一处必同步改测试，否则打回。

## 10. 落地与例外

### 10.1 新项目如何落地

本规范是通用底座。每个新项目在自己仓库写一份**《项目测试落地说明》**，填充本规范留白的占位项：

| 占位项            | 项目需明确                                                                       |
| ----------------- | -------------------------------------------------------------------------------- |
| 测试环境          | 测试 DB schema 名、缓存 db 编号、profile 名、连接方式                            |
| 目录约定          | 各层测试文件的存放目录与命名后缀                                                 |
| 跑测命令          | 各层测试的执行命令速查                                                           |
| 安全类清单        | 本项目《安全类代码清单》的位置与内容                                             |
| 覆盖率门槛        | 如与本规范默认值不同，列出项目定制值 + 理由                                      |
| 测试库重置与 seed | 测试库「清空 + 重新 seed」的具体机制（脚本 / fixture 位置）、seed 内容、维护方式 |
| 外部依赖          | 需 mock / 绕过的外部依赖清单及做法                                               |
| CI 集成           | CI 上各层何时触发、门禁何时卡红                                                  |

### 10.2 例外审批

- 任何**跳过某测试层**、**降低覆盖率门槛**、**提测时某层未跑**的情况，必须：
  1. 在自测报告中**显式说明**理由与影响范围；
  2. 经**技术负责人书面批准**。
- **未经批准的缺层 / 不达标，一律视为自测不通过**，测试人员准入审查时直接退回。
- 例外应是少数特例，不能成为常态。频繁申请同类例外，说明规范或工程实践需要调整——回写本规范。

### 10.3 规范演进

本规范是活文档。执行中发现新的坑、新的约定、新的工具实践，应回写本文档或对应的《项目测试落地说明》，保持规范与实践同步。

```markdown

### happy path

- C1 正常下单：合法商品 + 充足库存 → 返回订单号，订单状态=待支付

- C3 数量=1（最小值）→ 成功
- C4 数量=库存上限 → 成功，库存归零
- C5 金额为系统支持的最大值 → 成功，无溢出

- 副作用3：发"下单成功"消息 → C10 验消息发出
```
