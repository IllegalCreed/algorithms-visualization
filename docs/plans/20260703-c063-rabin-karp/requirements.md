## 背景

字符串大类首发 KMP（[[C-20260703-062]]）用**部分匹配表**加速匹配。本变更做 **Rabin-Karp**——另一种思路：**滚动哈希**。把长度 m 的窗口内容压成一个**哈希值**，每滑动一格用 O(1) 更新哈希（去掉首字符、加入尾字符），每个位置只比**一个数**而非逐字符；只有哈希相等时才逐字符**验证**（防哈希冲突误报）。它与 KMP 形成「哈希法 vs 前缀函数法」的对照，也是滚动哈希思想（后续可扩展到多模式、最长重复子串）的入门。

## 目标

1. 新增 **Rabin-Karp** 算法页（`/docs/rabin-karp`），接入既有 AlgorithmPlayer 框架（正文 + 分步播放器 + 四语言代码同步高亮）。
2. **复用 KmpView 字符串匹配轨**（第 2 消费者）+ 极小扩展 `windowStart`（窗口高亮）+ 空 LPS 时隐藏 π 行。零新轨。
3. 逐步呈现：每个窗口显示**窗口哈希 vs 模式哈希**；不等 → 跳过（滑动，快）；相等 → 逐字符**验证** → 命中/误报。哈希在变量面板/字幕展示。
4. 在「字符串」大类下新增第 2 个条目（菜单 + 首页网格）。

## 用户故事

- 作为学习者，我想看到 Rabin-Karp 每个窗口只比**一个哈希数**、滑动时哈希 O(1) 更新，理解它为何快。
- 作为学习者，我想看清「哈希相等还要逐字符验证」——哈希可能冲突（误报），验证才能确认真匹配。
- 作为学习者，我想对照 KMP（部分匹配表）与 Rabin-Karp（滚动哈希），体会字符串匹配的不同加速思路。

## 验收标准

- [ ] `/docs/rabin-karp` 正常渲染：介绍正文 + AlgorithmPlayer（字符串匹配轨 + 代码区）。
- [ ] 复用 KmpView：文本行（T=`abcabcab`）+ 模式串行（P=`cab`，对齐当前窗口）+ 窗口高亮（当前 m 格）；**无 π 行**（Rabin-Karp 不用 LPS）。
- [ ] 分步演示：`skip`（哈希不等，滑动）、`hashHit`（哈希相等）、`verify`（逐字符验证）、`found`（验证通过命中）；模式哈希 = 312、窗口哈希逐格显示。
- [ ] 命中：末态在 T 的下标 2、5 命中 P（`found` 含 2、5）。
- [ ] 菜单/首页「字符串」下出现「Rabin-Karp」；点击进入正确路由。
- [ ] KmpView 仅 additive 扩展 `windowStart`（KMP 不设 → 无 `.kmp-window`；KMP 有 LPS → π 行照常）；既有轨 + 全部算法页零回归。
- [ ] 门禁全绿（format/lint/type-check/单测含覆盖率/e2e），真机自检通过。

## 非目标

- 不做多项式哈希的严格防碰撞（双哈希/大素数）——演示用小模数、清晰数值；碰撞与验证在正文讲。
- 不做可变输入（文本/模式固定）。
- 不做 Boyer-Moore / AC 自动机 / Manacher——留作字符串大类后续。
