## 背景

M4 广度第六项 B6（进阶）。继线段树后新增**B 树 / B+ 树**独立页。backlog 点名「数据库 / 文件系统索引」。

**定位**：B 树是**多路平衡查找树**——一个节点装**多个 key + 多个孩子指针**，所有叶子在同一层，树又**矮又宽**。它专为**磁盘/数据库索引**而生：一次磁盘 IO 读一个节点（能装几十上百个 key），只要几层就能在海量数据里定位。**B+ 树**是它的改良：**数据全部只放叶子**，内部节点只当「路标」做路由，**叶子之间用链表串起来**——范围查询直接顺着叶链横扫。MySQL InnoDB 主键索引、文件系统目录索引用的都是 B+ 树。

**为何以 B+ 树为可视化主体**：B+ 树相对 B 树最大的优势就是「叶链范围查询」，因此一页能同时演示**两大看点**——「下钻查找」（B 树/B+ 树共有）+「叶链范围扫描」（B+ 树独有），内容比只讲 B 树更扎实、更贴近工程实际。

**新页 vs 加节**：广度项需 **4 处接线**——路由 + 侧边菜单 + 首页网格 + 图标（套路已跑通 5 次），外加 `Article/DataStructure/BTree.vue`。

## 三个地基决策

1. **固定 B+ 树**（仿图/字典树/并查集/跳表/线段树固定结构的稳妥做法，结构定死→可单测）：阶-5 B+ 树（每节点最多 4 key），**2 层**：
   - root（内部·路标）：routerKeys `[25, 45]`，3 个子指针。
   - 3 个叶子（存全部数据 + 叶链）：L0 `[5,10,15,20]`（< 25）、L1 `[25,30,35,40]`（25 ≤ x < 45）、L2 `[45,50,55,60]`（≥ 45）。
   - 叶子链表：L0 → L1 → L2。
   - 共 12 个数据 key、树高仅 2——对照 BST（12 个数要 4 层）凸显「矮胖、磁盘友好」。
2. **查找演「多路下钻」**：`search(target)` 从 root 按路标 key 选子区间，下钻到对应叶子，叶内判断含不含。可视化点亮**下钻路径**（root → 叶）+ 命中 key 格（绿），显「几层就到、一个节点筛掉一大批」。
3. **范围查演「叶链横扫」（B+ 树灵魂）**：`rangeScan(lo, hi)` 先定位 lo 所在叶，再**沿叶链向右**收集所有 lo ≤ k ≤ hi 的 key，遇到超过 hi 即停。可视化点亮**涉及的叶 + 命中 key 格（黄）+ 叶链箭头**，显「定位一次 + 顺序扫一段」。

## 与现有结构页的关系（一句话）

`BTreeViz` 复用「SVG 节点 + 边 + 走位点亮」画法（承袭 TreeViz/线段树），但节点是**多 key 的宽框**、查找是**多路下钻**、范围查是**叶链横扫**——讲二叉树都没讲的「多路平衡 + 磁盘索引 + 范围扫描」。
