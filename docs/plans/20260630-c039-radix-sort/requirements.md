## 背景

M4 数据结构全收官、M6 图算法开张（Dijkstra + Kruskal）后，Owner 指出**排序覆盖面少、很多细分没照顾到**，规划了 M7 排序细分扩充（`docs/plans/sorting-backlog.md`，S1–S9 三阶段）。**阶段一首发 S1 = 基数排序 Radix Sort**——它和计数排序、桶排序同属**非比较线性排序**，是当年首页里被注释掉、最该补的一个（线性排序三件套就差基数 + 桶）。

**定位**：基数排序不比较大小，而是**按「位」分配 + 收集**。LSD（从最低位起）：先按**个位**把所有数分到 0–9 共 10 个桶、再按桶序收集起来；再按**十位**分桶收集……处理完最高位后整体有序。每一轮分配 + 收集都是稳定的，多轮叠加得到正确顺序。线性复杂度 `O(d·(n+k))`（d=位数、k=10）。

**范式**：排序页**复用现成的「算法播放器」框架**（C-006）——多语言代码视图 + 行高亮 + 单步/播放 + 变量面板 + 可视化轨道。基数排序的「10 桶分配 + 收集」复用**计数排序的桶轨 `CountView`**（C-014 已建），与已有 8 个排序同范式、不新造画法。

## 三个地基决策

1. **固定输入数组**（仿所有可视化的固定数据 → 可测）：8 个两位数 `[42, 7, 25, 63, 18, 31, 56, 9]`（最大 63 → **2 轮**：个位、十位）。
   - 第 1 轮（按个位）：分桶 `1:[31] 2:[42] 3:[63] 5:[25] 6:[56] 7:[7] 8:[18] 9:[9]` → 收集 `[31,42,63,25,56,7,18,9]`。
   - 第 2 轮（按十位，单位数十位记 0）：分桶 `0:[7,9] 1:[18] 2:[25] 3:[31] 4:[42] 5:[56] 6:[63]` → 收集 `[7,9,18,25,31,42,56,63]` ✓ **有序**。
2. **LSD + 稳定分配**：从最低位到最高位；每轮按当前位把元素**依序**放入对应桶（保持稳定），再按桶 0→9 顺序收集回主数组。演示「为什么多轮分配能排好」。
3. **复用播放器 + CountView 桶轨**（与计数排序同范式，C-014 参照）：算法模块产出**逐步**步骤（数组快照 + 当前位 + 10 桶状态 + 指针/高亮 + 变量 + 当前代码行），视图接 `AlgorithmPlayer`，桶用 `CountView`（10 桶；若 CountView 仅支持 max+1 桶则按 0–9 配置或小幅适配，design 阶段定）。

## 不做什么（边界）

- **不做 MSD 基数排序 / 负数 / 任意进制 / 可编辑输入 / 字符串基数排序**：本页固定数组、LSD、十进制 2 轮主线；MSD、负数处理、桶排序（S2）由正文带过或后续。
- **不改算法播放器框架 / 既有 8 排序 / 数据结构 / 图算法**：仅新增 radix-sort 模块 + RadixSort.vue + 4 处接线 + 改排序 TC-HOOK 计数；CountView 如需适配仅做最小扩展且不破坏计数排序（design 评估，若改 CountView 要保证计数排序 Case 零回归）。
- **不一次补完线性排序**：本变更只做基数排序；桶排序（S2）后续。
